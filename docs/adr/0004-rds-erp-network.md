# ADR 0004 — RDS ERP networking choices

**Status**: Accepted
**Date**: 2026-05-01

## Context

Phase 3 stands up an RDS Postgres instance to simulate ACME's ERP source.
The operator wants to connect to the database from a local machine to load CSV
data (~530K rows in full mode) and run ad-hoc verification queries. The
ergonomic options for connectivity are:

1. RDS in public subnets, `publicly_accessible = true`, security group
   allowlists the operator's home IP
2. RDS in private subnets, connect via a bastion EC2 host
3. RDS in private subnets, connect via AWS Systems Manager Session Manager
   port-forwarding
4. RDS in private subnets, load data via Lambda or EC2 inside the VPC

## Decision

We use **option 1**: public subnets, `publicly_accessible = true`, ingress on
port 5432 restricted to the operator's `/32` IPv4 in the `acme-finance-dev-rds-sg`
security group.

## Why

- **Iteration speed.** `psql` and Python `psycopg.connect()` both work directly
  from the local machine — no extra hops, no SSM agent setup, no bastion
  instance to maintain.
- **No extra cost.** Bastion EC2 (~$3-5/mo) is unnecessary for a learning
  project where the operator is the only consumer.
- **Security is still acceptable.** A `/32` ingress rule means the public DNS
  endpoint accepts connections from one IP only. The master password is
  in Secrets Manager (32 chars, generated by Terraform). Storage is encrypted
  with the lake CMK. Backups are encrypted. The blast radius of a
  misconfiguration is bounded by KMS access.
- **Production teams who need this same pattern in dev typically use option 1
  with IP allowlists too** — it's the standard "developer dev DB" setup.

## Operator's IP changes

Residential ISPs rotate IPs occasionally (Comcast often does after a router
reboot). When that happens:

```bash
# Refresh the allowlist with the current IP
./scripts/refresh-my-ip.sh
```

This script updates `terraform.tfvars` with the new IP and re-runs
`terraform apply` (idempotent, ~30 seconds for the SG rule replacement).

## Hardening path (not implemented now, easy to switch later)

If we ever take this further than learning:

1. Switch `publicly_accessible = false` in `infra/modules/rds-erp/main.tf`.
2. Move RDS to `private_subnet_ids` instead of `public_subnet_ids` in
   `infra/envs/dev/main.tf`.
3. Add a Session Manager-enabled bastion via `aws_ssm_document` for
   port-forwarding. Or stand up a Lambda VPC handler for data loads.

The operator IP allowlist rule auto-becomes a no-op (cidr_blocks doesn't apply
when `publicly_accessible = false`).

## Consequences

- Loading data is fast (~3 minutes for 530K rows in full mode).
- The endpoint is in public DNS — `acme-erp-dev.<id>.us-east-1.rds.amazonaws.com`
  resolves from anywhere, but only allowlisted IPs can establish TCP.
- If the operator's IP changes, they lose access until they re-run
  `refresh-my-ip.sh`. Acceptable for a single-developer project.
